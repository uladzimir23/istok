# ADR-001 — PocketBase as Backend

> **SUPERSEDED 2026-05-01.** Решение пересмотрено в [[ADR-005]] до начала имплементации.
> Причина: для Phase 1 (миграция витрины с Tilda без онлайн-функционала) backend
> избыточен — каталог меняется редко, заявки идут в email+Telegram. PocketBase отложен
> в Phase 2 с зафиксированными trigger-условиями. Текст ниже сохранён как исторический
> контекст и ссылочный материал (плюсы/минусы вариантов A–E переиспользуем в ADR-005).

## Контекст

Мигрируем мебельную фабрику «Исток-мебель» с Tilda на собственный стек. Каталог
состоит из трёх направлений (~30 SKU суммарно, горизонт роста до ~100–200 SKU):

- Театральные кресла (B2B/госзаказ): 6 моделей (М1, М2, М3, М3-1, М3-Г, ПМ-1).
- Корпусная мебель: комоды, столы, стеллажи, шкафы.
- Детские кроватки «Элис» (суббренд ELIS-MEBEL): 8 моделей (Николь, Мелита, Паулина,
  Эльза, Доминика, Афина, Ясмина, Мишель).

Контент-команда на стороне клиента — 1–2 человека без технического бэкграунда. Им
нужна готовая, нетехническая админка для правок товаров, заявок и портфолио госзаказа.
Дополнительно нужны: хранилище файлов (фото товаров/проектов), auth редакторов, простой
деплой и надёжный бэкап. В reference-проекте `flex-glass` PocketBase уже использовался
успешно — есть переносимые паттерны и скилл `flex-glass/.claude/skills/pocketbase.md`.

## Варианты

### A. Sanity (managed headless CMS)
- Плюсы: быстрый старт, отличное Studio с превью, развитая экосистема, GROQ-запросы.
- Минусы: SaaS-зависимость, ценник растёт с трафиком/документами, generic UX (не
  доменно-заточенный под мебель), данные на стороннем хосте.

### B. Payload CMS (Node, self-host)
- Плюсы: TypeScript-first, удобные коллекции и hooks, self-host, гибкие ACL.
- Минусы: требует Node + Postgres/Mongo → больше инфраструктуры, чем нужно; медленнее
  старт; нет встроенного file storage.

### C. Custom admin (Next.js + Drizzle + Postgres + Auth.js)
- Плюсы: полный контроль над UX, доменно-заточенная админка под мебель.
- Минусы: 2–3 недели работы только на слой admin/auth/storage — прежде чем начать сайт.

### D. Content-as-code (MDX / TypeScript в репо)
- Плюсы: ноль инфраструктуры, контент в git, типобезопасность через Zod.
- Минусы: контент-менеджер фабрики не умеет в git/PR. Каждая правка цены = задача
  разработчику. Неприемлемо при ≥30 SKU и регулярных обновлениях.

### E. PocketBase (single Go binary, SQLite) — выбранный
- Плюсы: всё в одном (БД + стоковая админка + auth + file storage + REST API + JS SDK +
  realtime), single binary без дополнительных зависимостей, бэкап = `cp pb_data.db`,
  деплой на любой VPS, схема под git через `pb_migrations/*.js`, бесплатный self-host,
  есть переносимый опыт из `flex-glass`.
- Минусы: SQLite — один сервер (нет горизонтального масштабирования), стоковая админка
  не доменно-заточена под мебель (кастомизируема через collection-config и hooks), realtime
  в Docker требует аккуратной настройки.

## Решение

Выбираем **E. PocketBase**.

Объём каталога (≤200 SKU + заявки + портфолио госзаказа) укладывается в SQLite на один
сервер с колоссальным запасом — SQLite спокойно держит миллионы строк. Готовая стоковая
админка экономит 2–3 недели разработки в Этапе 1, при этом её достаточно редакторам
фабрики на старте. Single-binary деплой и `cp pb_data.db` для бэкапов соответствуют
принципу «лучшая практика = простая практика», который этот проект должен демонстрировать
как эталон.

Если упрёмся в ограничения SQLite (горизонтальный scale, тяжёлый full-text search,
специфичные ACL) — данные экспортируются из SQLite в Postgres за 1–2 дня (`pgloader` или
скрипт). Кастомные экраны (например, конфигуратор сборок размер×цвет×опции) добавляются
позже как Next.js-страницы поверх PocketBase API — без изменения базовой инфраструктуры.

## Последствия

**Что делаем:**
- Поднимаем PocketBase в Docker; схема описывается в `pocketbase/pb_migrations/*.js`
  и хранится в git.
- Next.js (frontend) обращается к PocketBase через JS SDK с серверной стороны
  (App Router server components / route handlers).
- ISR + `revalidateTag` от PocketBase webhook (`pb_hooks/onUpdate.pb.js`) при изменении
  товара или коллекции.
- File storage — встроенный в PocketBase (локальная FS; при росте трафика переходим на
  S3-совместимый bucket — решим отдельным ADR).
- Auth редакторов — встроенный admin/users в PocketBase (логин + пароль, опционально
  magic link через SMTP).
- Бэкапы — cron-снапшот `pb_data/` в зашифрованный S3-bucket (детали — ADR по бэкапам).
- Локальная разработка — `docker compose up`, админка на `http://localhost:8090/_/`.
- Адаптируем скилл `flex-glass/.claude/skills/pocketbase.md` в
  `istok/.claude/skills/pocketbase.md`.

**Что НЕ делаем:**
- Не запускаем несколько инстансов PocketBase параллельно (SQLite не поддерживает
  горизонтальный scale).
- Не используем PocketBase для тяжёлых фоновых вычислений / очередей задач — для этого
  появится отдельный сервис при необходимости.
- Не строим кастомную доменную админку в Этапе 1 — используем стоковую.

**Обратимость: средняя.**
- Данные: SQLite-экспорт → Postgres через `pgloader` или скрипт — 1–2 дня при текущем
  объёме.
- Схема: переписать в Drizzle/Prisma — ~1 день для текущего масштаба.
- Frontend: Next.js легко переключить на другой data-source через единый адаптер
  `packages/data-source/`.
- Лок-ин — только привычки стоковой админки у редакторов. Реальный технический лок-ин
  минимален.

**Кто затронут:**
- Разработка — изучение паттернов PocketBase: collection-rules, hooks, JS migrations
  (есть донор `flex-glass`).
- Контент-команда фабрики — обучение стоковой админке PocketBase (1 короткий playbook +
  ~30 минут демо).
- DevOps — поднять Docker на VPS, настроить cron-бэкап, certbot для HTTPS.

## Имплементация

Не начата. Следующие шаги после принятия ADR-002 (фронтенд) и ADR-005 (хостинг):
1. Создать `pocketbase/` в корне репо, `pb_migrations/`, `pb_hooks/`.
2. Добавить `docker-compose.yml` с сервисом PocketBase.
3. Адаптировать `pocketbase.md` скилл из `flex-glass`.
4. Завести коллекции: `products`, `categories`, `leads`, `portfolio_projects`.
