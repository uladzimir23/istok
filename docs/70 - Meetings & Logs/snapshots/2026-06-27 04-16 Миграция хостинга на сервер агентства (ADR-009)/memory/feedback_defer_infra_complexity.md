---
name: Defer infra complexity until justified
description: Пользователь явно проявил установку: не добавлять инфраструктурные слои до тех пор, пока не появится конкретный триггер из бизнеса.
type: feedback
---

Принцип зафиксирован через ADR-005 (2026-05-01): когда ADR-001 (PocketBase) был готов и
одобрен, пользователь принял решение отложить backend до Phase 2 — потому что для Phase 1
(миграция витрины) он избыточен.

**Паттерн поведения:** пользователь предпочитает начать с минимальной инфраструктурой
(content-as-code, flat repo, single VPS) и добавлять сложность только при наступлении
конкретного бизнес-триггера. Не «на вырост», а «когда потребуется».

**Применение в этом проекте:**
- Phase 1 — content-as-code (MDX/TS + Zod), без CMS и без backend.
- Phase 2 — PocketBase добавляется только если: второй редактор без git, правки чаще
  раза в неделю, нужен инбокс/CRM, sync с маркетплейсами, онлайн-заказ с конфигуратором,
  или личный кабинет/роли.
- Структура репо — плоская (ADR-007), без monorepo. PocketBase в Phase 2 = новый сервис
  в docker-compose, не workspace.

**Как применять в других проектах:** при старте нового проекта НЕ предлагать «полный стек»
сразу. Сначала — минимум, достаточный для Phase 1. Сложность — по триггерам.
