# 55 — Design System

> **Зачем эта секция.** Дизайн-система — это технический слой между брендом
> (`docs/65 - Brand/`) и кодом (`web/src/shared/design-system/`). Здесь живут
> референсы, реестр компонентов, дискуссии по токенам, архитектурные решения
> по DS, которые не дотягивают до уровня ADR. Из этой секции рождаются ADR
> (например, ADR-004 SCSS Modules) и финальные значения токенов.

## Разделение с другими секциями

| Секция | Что там | Уровень изменчивости |
| --- | --- | --- |
| `docs/65 - Brand/` | Бренд-айдентика: логотипы, голос, фотостиль, identity | Стабильно, редко правится |
| **`docs/55 - Design System/` (эта)** | **Технические референсы, реестр компонентов, обсуждение токенов** | **Итерируется в процессе** |
| `docs/40 - Architecture/42 - ADR/` | Архитектурные решения (ADR-004 SCSS modules и др.) | Immutable после accepted |
| `web/src/shared/design-system/` | Реализация токенов и компонентов в коде | Развивается с фичами |

## Что внутри

- **`Tilda reference.md`** — извлечённые палитра, типографика, IA, паттерны со
  старого сайта `istokmebel.by` (Tilda export). Главный «опорный материал»
  по визуалу для нового сайта — клиент хочет сохранить узнаваемость.
- **`Inspiration & Mood.md`** — сторонние референсы (другие мебельные сайты,
  B2B-фабрики, детские бренды). Открытая доска идей.
- **`Component Inventory.md`** — реестр компонентов сайта: что нужно сверстать
  (header, RoomCard аналог, Hero, Gallery, ContactForm, ...). Статус каждого:
  планируется / в работе / готов.
- **`Design tokens decision.md`** — финальная резолюция токенов (палитра, шрифты,
  spacing, radius, motion) перед началом имплементации `web/src/shared/design-system/`.
  Превращается в ADR при принятии.

## Workflow

1. Новые наблюдения → дописываем в соответствующий документ (свободно).
2. Когда есть устоявшаяся развилка — фиксируем в `Design tokens decision.md`.
3. Когда финализирована система токенов — оформляем как ADR-NNN или updates ADR-004.
4. Имплементируем в `web/src/shared/design-system/`.
