# Skill: Documentation Writing

## Формат файлов

- `.md` — по умолчанию для всего (бриф, аудит, ADR, session log).
- `.mdx` — позже, если поднимем docs-site с React-компонентами (Callout, Tag и т.п.).
  Пока всё в чистом markdown.

## Структура файла

```markdown
---
title: Заголовок документа
description: Короткое описание (для будущих frontmatter-аггрегаторов)
date: 2026-05-01
order: 10
tags: [audit, brand]
status: draft | in-review | final
---

# Заголовок

Первый параграф — **TL;DR в 1-2 предложениях**. Если читатель дальше не пойдёт,
должен вынести главное.

## Далее — по секциям
```

### Обязательные frontmatter-поля

- `title` — заголовок страницы.
- `description` — короткое описание.
- `order` — порядок внутри родительской папки (меньше = выше).

### Опциональные

- `date`, `tags`, `status`, `category`, `audience` (для dept-линз).

## Johnny Decimal структура

```
docs/
├── 00 - Indexes/                    # Dashboard, dept-линзы, Архитектура знаний
├── 10 - Brief & Requirements/       # бриф, доступы, ТЗ
├── 20 - Audit/                      # аудит as-is
├── 30 - SEO/                        # стратегия SEO
├── 40 - Architecture/
│   └── 42 - ADR/                    # Architecture Decision Records
├── 45 - Engineering Workflow/       # branching, commits, PR, release
├── 50 - Roadmap/                    # phase planы
├── 60 - Content/                    # тексты, описания моделей
├── 65 - Brand/                      # логотипы, палитра, типографика
├── 67 - SMM/                        # контент-план соцсетей
├── 70 - Meetings & Logs/            # session logs + snapshots
├── 80 - Templates/                  # шаблоны для новых документов
├── 90 - Ideas & Backlog/            # обсуждение идей без решения
├── 95 - Attachments/                # PDF, скриншоты, исходные материалы
└── 97 - Reports/                    # отчёты для клиента
```

**Правило**: номер категории — это URL-префикс (когда появится docs-site). Префикс
`NN - ` в URL **отбрасывается**.

## Wiki-links (Obsidian-style)

```markdown
См. [[Бриф проекта]] — работает как ссылка на `docs/10 - Brief & Requirements/Бриф проекта.md`.

Если одноимённые файлы в разных папках — уточняем: [[20 - Audit/Сайт as-is]].
```

Obsidian сам найдёт файл по filename. Backlinks — бесплатно (граф знаний).

## Тон и стиль

### Язык — русский (основной)

Документация для клиента-русскоязычного и для собственной команды. Исключение:
технические термины (PocketBase, Next.js, schema.org, webhook) — оставляем как есть.

### Стиль — «аналитик объясняет владельцу бизнеса»

- Короткие предложения. Длинное разбивать.
- Сначала вывод, потом обоснование.
- Конкретика вместо абстракций: не «нет аналитики», а «ни Метрики, ни GA4 на главной —
  серверный лог не покажет, откуда пришёл пользователь».
- Цифры и цитаты из источников, а не утверждения.
- Избегать «мы думаем», «возможно», «наверное» — либо факт, либо гипотеза с пометкой.

### TL;DR обязательно

На любой странице длиннее ~300 слов — первый параграф после заголовка это TL;DR на
1-3 предложения.

### Нумерация фактов / пруфов

Для отчётов — нумеруем: «Факт 1», «Факт 2», «Пруф 1». Читателю легче ссылаться в
обсуждении.

## Чек-лист перед коммитом документа

1. [ ] Frontmatter на месте (title, description, order).
2. [ ] Нет реальных секретов (см. `redaction.md`).
3. [ ] Wiki-links работают (целевые файлы существуют).
4. [ ] Минимум 2 ссылки в секции «Связанные» (см. `knowledge-graph.md`).
5. [ ] TL;DR в начале (если > 300 слов).
