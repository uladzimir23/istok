# Skill: Knowledge Graph

## Принцип

Документация — это **граф**, а не коллекция файлов.

- Узел = документ (ADR, session log, dashboard, audit, brief, план).
- Ребро = wiki-link (`[[Имя документа]]`).
- Obsidian делает backlinks автоматически — двунаправленность бесплатна.

**Цель:** от любого узла человек доходит до любого другого **за ≤3 перехода**.
Если где-то нужно 5–7 переходов или приходится искать через файловый поиск —
граф дырявый, надо прошить.

## Обязательная секция «Связанные»

Каждый значимый документ заканчивается секцией:

```markdown
## Связанные

- [[ADR-001 PocketBase as backend]] — комплементарное решение
- [[2026-05-01 03-30 SDD init]] — сессия, в которой создано
- [[Dashboard]] — общий статус
```

**Правила:**
- Минимум **2 ссылки**. Меньше — признак orphan-узла.
- Каждая ссылка с **краткой аннотацией** (3–7 слов).
- Ссылки бывают 3 типов (см. ниже) — в хорошем документе представлены хотя бы 2.

## Типы ссылок

| Тип | Что значит | Пример |
|---|---|---|
| **Контекст** | Почему эта работа вообще началась | `[[Бриф проекта]]`, session log с решением |
| **Соседи** | Связанные по смыслу узлы того же уровня | `[[ADR-001]]` ↔ `[[ADR-002]]`, прошлая/следующая сессия |
| **Хабы** | Индексы, куда узел «впадает» | `[[Dashboard]]`, `[[01 Маркетинг]]`, `[[02 Разработка]]`, `[[Архитектура знаний]]` |

Документ без хотя бы одной хаб-ссылки теряется. Каждый узел должен «впадать» куда-то
вверх.

## Правила по типам документов

### ADR (`docs/40 - Architecture/42 - ADR/ADR-NNN *.md`)

- **Связанные ADR** — предыдущий/следующий по теме, комплементарные, отвергающие.
- **Session log**, в котором решение обсуждалось.
- Если ADR возник из аудита/брифа — ссылка на этот документ.

### Session log (`docs/70 - Meetings & Logs/*.md`)

- **Предыдущий session log** — цепочка времени.
- **Dashboard** — живой статус.
- **Все ADR**, созданные в этой сессии (если были).

### Dashboard / dept-dashboards

- Ссылается на все ключевые документы (ADR, планы, session logs).
- Каждый документ, важный для аудитории дашборда, должен быть упомянут.
- Правится через sub-агент `docs-sync`.

### Brief / Audit / Plan

- На Dashboard.
- На связанные ADR (если решения уже приняты).
- На session log, где обсуждалось.

## Antipatterns

- ❌ **Orphan-узел**. Документ создан, но никто на него не ссылается. В Obsidian Graph
  View такие видны как изолированные точки.
- ❌ **Слепые ссылки**: `См. [[ADR-001]]` без пояснения.
- ❌ **Хаб без backlinks**. Документ ссылается на Dashboard, но Dashboard про него не
  знает. Либо документ не важен (ссылка лишняя), либо в Dashboard дырка (добавить).
- ❌ **Только soft-backlinks**. Полагаться только на Obsidian backlinks panel рискованно —
  она требует открытия заметки. Явные «Связанные» лучше.

## Workflow при создании нового документа

1. Перед сохранением — открыть Dashboard + релевантный dept-dashboard и добавить ссылку
   на новый документ.
2. В новом документе — секция «Связанные» с 2–4 ссылками по типам выше.
3. Найти 1–2 существующих узла, где следует добавить встречную ссылку.
4. Проверить результат в Obsidian Graph View — новый узел не должен быть изолирован.

## Workflow при редактировании

Если добавилась новая связь в одну сторону — **всегда добавляем встречную** в целевой
документ. Односторонних рёбер не оставляем.
