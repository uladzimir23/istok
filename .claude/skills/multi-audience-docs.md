# Skill: Department Lenses (мульти-аудиторная документация)

## Зачем

Клиент (фабрика) — не однородная аудитория. У `istokmebel.by` минимум три угла зрения:

- **Маркетинг и продажи** — что мы запускаем, какие воронки (B2B-кресла, B2B-корпусная,
  B2C-Элис), какие KPI, когда можно лить трафик. Аудитория: отдел продаж (Анна),
  владелец, маркетолог (если появится).
- **Разработка** — стек, ADR-карта, зоны кода, блокеры, окружения, риски. Аудитория:
  мы + потенциально подрядчик клиента.
- **Производство / контент-команда** (когда понадобится) — как заводить новую модель
  в каталог, как делать рендеры, как обновлять цены и наличие.

Каждая получает **свой dashboard** в `docs/00 - Indexes/`, рассказывающий одну правду
на их языке. Источник истины один (`docs/40 - Architecture/`, ADR, brief, audit) —
дашборды это **линзы**.

## Принцип

```
            ┌───────────────────────────────────┐
            │  ИСТОЧНИК ИСТИНЫ (single)         │
            │  ADR, Roadmap, Brief, Audit       │
            └────────────────┬──────────────────┘
                             │
            ┌────────────────┼────────────────┐
            ▼                ▼                ▼
     ┌────────────┐   ┌────────────┐   ┌────────────┐
     │ Маркетинг  │   │ Разработка │   │ Производство│
     │ дашборд    │   │ дашборд    │   │ дашборд    │
     │            │   │            │   │            │
     │ бизнес-яз. │   │ техн. язык │   │ операц.    │
     │ KPI, даты  │   │ ADR, риски │   │ инструкции │
     └────────────┘   └────────────┘   └────────────┘
```

**Правило:** дашборды — это **проекции**, а не дублирование. Они ссылаются на источник
через wiki-links, не копируют его содержимое.

## Когда заводить новый dept-дашборд

- Появилась стабильная аудитория со своим жаргоном и критериями успеха.
- В обычном `Dashboard.md` им неудобно — он смешивает технику и бизнес.
- Они задают одни и те же вопросы.

**Не заводить ради:**
- Одноразового рассказа (это session log).
- Одного человека внутри уже существующей роли.
- «Чтоб красиво» (накладные на синхронизацию).

## Структура dept-дашборда

Шаблон лежит в `docs/80 - Templates/dept-dashboard.md`. Минимум:

1. **Что мы делаем для вас** — 2–3 предложения на их языке.
2. **Текущий приоритет** — что в работе, что ждёт.
3. **Deliverables / Зоны** — что они получат / увидят.
4. **KPI / критерии** — на что смотрят.
5. **Что нужно от них** — блокеры, доступы, контент.
6. **График / даты** — следующий milestone.
7. **В двух словах техническое** (для не-технических) — мостик.
8. **Глоссарий** (для не-технических) — расшифровка терминов.
9. **Где смотреть детали** — wiki-links на источник истины.

## Перевод тех-языка → бизнес-языка (для маркетинг-дашборда)

| Тех-язык (источник)                         | Бизнес-язык (маркетинг)                                       |
|---------------------------------------------|---------------------------------------------------------------|
| PocketBase + Next.js 15 SSR                 | Сайт умеет принимать заявки, цены и каталог обновляются за 5 минут |
| schema.org Product + ItemList               | Google и Yandex видят товары как карточки с ценой и наличием  |
| Яндекс.Метрика цели + Webvisor              | Видим, кто что смотрит, докуда доходит и где отваливается     |
| Cloudflare R2 + next/image                  | Фото грузятся быстро, не съедают мобильный трафик клиентов    |
| App Router SSG + ISR                        | Изменил цену в админке — через 60 секунд она на сайте         |
| Cutover istokmebel.by → новый стек          | В день X новый сайт встаёт на основной адрес, Tilda убираем   |

**Правило:** если фраза начинается с английского названия библиотеки или содержит
snake_case — она НЕ для маркетинг-дашборда.

## Перевод бизнес-задачи → тех-задачи (для дашборда разработки)

| Бизнес-формулировка                          | Тех-формулировка                                              |
|----------------------------------------------|---------------------------------------------------------------|
| «Чтоб заявка дошла Анне»                     | `app/api/inquiry` → Telegram Bot + Resend, ретраи + лог       |
| «Покажите кресла для тендеров»               | `/chairs` с ItemList schema, отдельный CTA «Запросить КП»     |
| «Цены меняем сами»                           | PocketBase admin → коллекция `products` → `price_byn` поле    |
| «Хотим аналитику»                            | Метрика-цели (просмотр товара, отправка формы), Webvisor      |

## Anti-patterns

- ❌ **Технический жаргон в маркетинговом дашборде** без расшифровки.
- ❌ **Бизнес-чеклист «надо принести фото»** в дашборде разработки — это блокер маркетинга.
- ❌ **Копирование больших таблиц** из источника в дашборд — лучше wiki-link.
- ❌ **Дашборд как «ещё один TODO»** — TODO живёт в Roadmap или трекере, дашборд это
  статус и навигация.
- ❌ **Расхождение фактов между линзами** — если в маркетинг-дашборде «запускаем 25-го»,
  а в разработческом «релиз 28-го» — кто-то лжёт.

- `05 Финансы.md` — счета, договоры, юр-владение доменом.

**Нумерация:** `00 Dashboard.md` — общий, `01..` — отделы в порядке приоритета для
проекта (маркетинг сейчас #1, потому что цель — продажи).
